Create a file named PRD.md for the project “VoteWise India”.

Write a complete Product Requirements Document for a hackathon project.

Project summary:
VoteWise India is an accessibility-first, AI-powered election process education platform for Indian users. It helps first-time voters and general citizens understand the election process in a simple, neutral, practical, and non-partisan way using guided learning, an educational assistant, quizzes, myth-vs-fact content, and accessible support features.

The PRD must be detailed, professional, realistic, and optimized for hackathon evaluation.

Required sections:

1. Product Title
- Project name
- One-line tagline

2. Product Vision
- Explain the long-term idea of the platform
- Emphasize civic education, accessibility, trust, and usability
- Mention that the platform is educational and non-partisan

3. Problem Statement
- Describe the real-world problem clearly
- Focus on confusion around voter eligibility, registration, documents, voting day process, EVM/VVPAT, and election literacy
- Explain why first-time voters and many general users may feel overwhelmed or uninformed

4. Target Users
Include user groups such as:
- first-time voter
- student
- working professional
- senior citizen
- users with accessibility needs
For each, briefly explain their needs

5. Goals
List the product goals clearly, such as:
- simplify election process education
- improve confidence for first-time voters
- provide accessible civic education
- use AI to personalize and simplify guidance
- provide a neutral learning experience
- create a practical and modern learning platform

6. Non-Goals
Include what the product does NOT do, such as:
- does not provide political recommendations
- does not suggest parties or candidates
- does not replace official election commission resources
- does not offer legal advice
- does not collect unnecessary user data

7. Core Features
Describe each feature in detail:
- home page
- personalized learning journey
- election process timeline
- AI educational assistant
- quiz and readiness tracker
- myth vs fact
- accessibility controls
- Google Maps resource locator
- disclaimer and help guidance

8. User Journeys
Write at least 4 realistic user journeys, such as:
- first-time voter trying to understand the process
- student wanting a quick learning path
- senior citizen needing larger text and simple guidance
- user asking the assistant about EVM or voting day
For each user journey, show step-by-step interaction

9. Functional Requirements
Write detailed functional requirements for the app
Include page-level and feature-level requirements
Mention logic for personalization, assistant, quiz scoring, accessibility controls, and resource locator

10. Non-Functional Requirements
Include:
- performance
- responsiveness
- maintainability
- scalability for a hackathon prototype
- accessibility-first experience
- secure input handling
- readable clean architecture

11. Accessibility Requirements
Include:
- semantic HTML
- keyboard navigation
- ARIA labels
- contrast support
- text resizing
- reduced motion
- responsive readable typography
- inclusive onboarding and content presentation

12. Security Requirements
Include:
- no exposed secrets
- environment variable usage
- input validation
- safe handling of assistant prompts
- safe fallback responses
- no unnecessary storage of user data
- non-partisan safe content boundaries

13. Google Services Requirements
Explain meaningful integration of:
- Gemini API for educational assistant and simplified explanations
- Google Maps for resource/help locator
Explain why these are meaningful and useful

14. Testing Requirements
Define testing expectations in detail:
- unit tests
- component tests
- input validation tests
- personalized path logic tests
- assistant fallback tests
- quiz scoring tests
- accessibility-related tests
- manual QA checklist

15. Success Criteria
Define what success looks like, especially aligned with judging criteria:
- polished UI
- clean code
- safe AI use
- test coverage
- strong accessibility
- meaningful Google services integration
- practical user value

16. Risks and Mitigations
Mention realistic risks:
- API failure
- misinformation risk
- accessibility oversights
- over-complexity
- vague content
- performance issues
Give mitigation strategies

17. Future Scope
Mention enhancements such as:
- more languages
- live official data integration
- speech-based assistant
- location-specific official resources
- richer quizzes and analytics

18. Disclaimer
Include a final statement clarifying:
- educational use only
- neutral/non-partisan
- not legal advice
- official sources should be consulted when necessary

Tone requirements:
- highly professional
- structured
- practical
- product-focused
- evaluator-friendly
- not too vague
- not too academic
